## 1. Problem Statement

### 現況痛點

台灣銀行業客服中心每日承受大量重複性查詢，主要問題如下：

| 問題 | 影響 |
|------|------|
| 60-70% 來電為一般性查詢（費率、流程、說明）| 人力成本高，客服人員無法專注複雜案件 |
| 平均等待時間 5-15 分鐘 | 用戶體驗差，放棄率高 |
| 非辦公時間（晚上、假日）無法服務 | 24/7 需求無法滿足 |
| 未解決問題缺乏系統化記錄 | 難以識別知識庫缺口，無法持續優化 |

### 機會

AI 客服助理可自動處理 80% 以上的標準化查詢，讓真人客服專注於需要判斷力的複雜案件（爭議款項、特殊申請、情緒處理），同時提供 24/7 全天候服務。

## 2. Business Goals

| 目標 ID | 目標描述 | 衡量指標 |
|---------|----------|---------|
| **BG-01** | 降低客服電話量 30% | 每月來電量趨勢 |
| **BG-02** | 提升第一次解決率（FCR）至 80% 以上 | 用戶回饋 + Fallback 率 |
| **BG-03** | 提供 24/7 全天候服務，無額外人力成本 | 系統可用率 > 99% |
| **BG-04** | 建立數據驅動的 AI 持續優化機制 | 每月知識庫更新頻率 |

## 3. Scope

### In Scope（本版本包含）

- FAQ 自然語言查詢（一般性銀行問題）
- 登入後帳務查詢（帳單、餘額、交易明細）
- 信用卡掛失引導流程
- 未解決問題自動記錄與統計報表
- 中文、英文雙語支援
- 真人客服轉接

### Out of Scope（本版本不包含）

- 真實銀行後台系統整合（採用 Mock API）
- 轉帳、匯款等金融操作執行
- 個人化推薦（理財建議、產品推薦）
- 語音介面
- 行動 APP

## 5. Use Cases（三個 MVP）

### UC-01：FAQ 查詢（RAG Mode）

| 欄位 | 內容 |
|------|------|
| **Use Case ID** | UC-01 |
| **名稱** | 一般性銀行 FAQ 查詢 |
| **Actor** | 未登入 / 已登入用戶 |
| **AI 工作模式** | RAG Mode（從知識庫檢索後生成回答） |
| **前置條件** | 用戶輸入問題 |
| **後置條件** | 用戶得到正確資訊 或 收到 Fallback 說明 |

**正常流程（Happy Path）**：
1. 用戶輸入問題（中文或英文）
2. 系統偵測語言，路由至對應知識庫
3. FAISS 搜尋 Top-3 相關文件 chunks
4. Similarity score ≥ 0.7 → Claude 根據 context 生成回答
5. 回答附上「參考來源」文件名稱
6. 涉及利率/費用時自動附加免責聲明

**替代流程（Alternative Path）**：
- Similarity score < 0.7 → Fallback：誠實告知無法回答，提供客服電話
- 非銀行相關問題 → Safety Guard 攔截，友善回導

**查詢範例**：
- 「信用卡年費多少？如何申請免年費？」
- 「ATM 每日提款上限是多少？」
- "What documents do I need to open a savings account?"
- 「房貸利率現在是多少？」

---

### UC-02：帳務查詢（API Mode）

| 欄位 | 內容 |
|------|------|
| **Use Case ID** | UC-02 |
| **名稱** | 個人帳務查詢 |
| **Actor** | 已登入用戶 |
| **AI 工作模式** | API Mode（呼叫後端 API 取得個人資料） |
| **前置條件** | 用戶已完成登入驗證 |
| **後置條件** | 用戶得到個人帳務資訊 |

**正常流程（Happy Path）**：
1. 用戶輸入帳務查詢（「我本月帳單多少？」）
2. 系統確認登入狀態 → 已登入則繼續
3. 呼叫 Mock Banking API 取得資料
4. Claude 將 JSON 資料整理成自然語言回覆
5. 詢問是否需要進一步查詢

**替代流程（Alternative Path）**：
- 用戶未登入 → 提示先登入，提供登入連結
- API 呼叫失敗 → 告知系統暫時無法存取，建議改用行動 APP

**查詢範例**：
- 「我本月信用卡應繳金額是多少？」
- 「上個月有哪些超過 $5,000 的消費？」
- 「我的活存帳戶餘額是多少？」

### UC-03：信用卡掛失（Workflow Mode）

| 欄位 | 內容 |
|------|------|
| **Use Case ID** | UC-03 |
| **名稱** | 信用卡遺失掛失流程 |
| **Actor** | 已登入或未登入用戶（未登入時需先驗證身分） |
| **AI 工作模式** | Workflow Mode（多步驟引導，含 Tool 呼叫） |
| **前置條件** | 用戶回報信用卡遺失或被盜 |
| **後置條件** | 卡片成功掛失 + 服務單建立 |

**正常流程（Happy Path）**：

| 步驟 | AI 行為 | 用戶行為 |
|------|---------|---------|
| Step 1 | 詢問身分後四碼確認身分 | 提供正確資訊 |
| Step 2 | 列出持有卡片，詢問是哪張遺失 | 確認卡片 |
| Step 3 | 再次確認「確定要掛失此卡片嗎？」| 確認執行 |
| Step 4 | 執行掛失（呼叫 block_card API）| — |
| Step 5 | 建立服務單，提供 Ticket ID | — |
| Step 6 | 說明後續補發流程，詢問是否需要轉真人 | 選擇結束或轉接 |

**替代流程（Alternative Path）**：
- 身分驗證失敗 3 次 → 自動觸發 Human Handoff
- 用戶中途說「取消」→ 終止流程，提供客服電話
- 用戶要求立即轉真人 → 提供 Human Handoff

## 8. Human Handoff 觸發條件

| 觸發條件 | 偵測方式 |
|----------|---------|
| 用戶輸入含「投訴」「申訴」「我要告你」"complaint" "speak to human" | 關鍵字比對 |
| 用戶輸入含「真人」「客服人員」"agent" "representative" | 關鍵字比對 |
| 連續 3 次回答被用戶否定（「不對」「不是這個」「沒用」）| Fallback counter |
| 信用卡掛失流程中身分驗證失敗 3 次 | Workflow state |
| 用戶在 Workflow 中明確要求中止 | 明確意圖 |

**Human Handoff 回應範本**：
```
為您轉接客服人員協助。

📞 客服電話：0800-XXX-XXX
🕐 服務時間：週一至週五 09:00-17:00

非服務時間您可：
• 留下問題，客服將於下個工作日與您聯繫
• 使用行動 APP 的自助服務功能

已為您建立服務單 #T-{ticket_id}，以便客服人員了解您的情況。
```

## 9. Disclaimer 設計（責任邊界）

| 層次 | 實作位置 | 內容 |
|------|---------|------|
| **UI 層** | 聊天視窗底部固定顯示 | 「本助理提供之資訊僅供參考，不構成財務建議。實際利率、費用以官網公告為準。」 |
| **Prompt 層** | System Prompt 規則 | 凡涉及利率、費用、法律責任的回答，結尾必須附加免責聲明 |
| **Intent 層** | `loan_info`、`interest_rate` intent 觸發時 | 強制插入 disclaimer，不依賴 AI 自行判斷 |

## 11. 已知限制與風險

| 風險 | 說明 | 緩解策略 |
|------|------|---------|
| AI 幻覺 | Claude 可能回答不在知識庫的資訊 | Prompt 限制只根據 context 回答；Similarity threshold 過濾；顯示來源文件 |
| 知識庫缺口 | 用戶問到知識庫未涵蓋的問題 | Fallback 記錄系統 + 每週人工審查補充 |
| 語言混用 | 中英文夾雜的輸入（如 Chinglish）| 以主要語言判定；Claude prompt 容錯處理 |
| Mock API 限制 | 資料為假資料，不代表真實銀行狀態 | 所有 Mock 資料有明確標示；面試時說明替換真實 API 的方法 |
| 單用戶 Demo 限制 | 本版本未設計多用戶並發 | 面試時說明生產環境的 Rate Limiting 與快取策略 |

## 12. 術語表（Glossary）

| 術語 | 說明 |
|------|------|
| **RAG** | Retrieval-Augmented Generation，從知識庫檢索相關內容後再生成回答 |
| **FAISS** | Facebook AI Similarity Search，向量相似度搜尋函式庫 |
| **Fallback** | 機器人無法回答時的替代處理（告知用戶、記錄問題）|
| **Human Handoff** | 將對話轉交給真人客服的機制 |
| **Intent** | 用戶輸入的意圖分類（如：查帳單、問費用、掛失）|
| **Similarity Threshold** | 向量搜尋的相似度門檻，低於此值判定為 Fallback |
| **Mock API** | 模擬真實銀行後台 API 的假資料服務 |
| **Ticket** | 客服服務單，用於追蹤未解決的問題 |
| **ADR** | Architecture Decision Record，記錄技術選型決策及理由 |
